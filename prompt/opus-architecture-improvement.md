# 프로젝트 아키텍처 및 모노레포 고도화 (Opus 전용 프롬프트)

> **컨텍스트 최적화 지침** 
> 이 문서와 하단에 지정된 참조 파일들만 우선 읽고 프로젝트 구조를 파악하세요. 전체 비즈니스 로직을 처음부터 탐색할 필요는 없습니다. 
> 우리의 프로젝트는 하위 `apps/api`(Spring Boot 4.0), `apps/web`(React 19) 폴더를 가진 기초적인 구조의 모노레포입니다.

## 💡 최고 수준 AI 에이전트(Opus)를 위한 자율 판단 가이드
아래 제시된 4가지 주요 과제는 **과거 분석을 바탕으로 사용자가 도출한 "기본 아이디어 초안"일 뿐**입니다. 
당신이 프로젝트 파일(`Makefile`, `docker-compose.yml`, `CLAUDE.md` 등)을 직접 분석한 후, **"제안된 방법보다 아키텍처 설계 관점에서 더 나은 방법론, 트렌디한 도구, 혹은 더 효율적인 코드 레벨 패턴이 있다"고 판단되면 주저 없이 본 제안을 폐기하고 더 나은 솔루션으로 대체해서 설계**해 주세요.
당신의 최고 수준 소프트웨어 아키텍처 및 엔지니어링 지식을 전적으로 신뢰하며 전권을 위임합니다.

### ⚠️ 핵심 제약 조건: "유지보수성"과 "오버엔지니어링 방지"
신기술이나 복잡한 인프라를 무작정 도입하는 것을 경계해 주세요. 
새로운 기술(예: Kafka, MSA 전면 개편 등)을 도입할 경우, **그로 인해 얻는 이점이 프로젝트의 '복잡성 증대(Complexity)'와 '유지보수비용 증가(Maintenance Cost)'를 감수할 만큼 충분히 가치 있는지 엄격하게 평가**하세요. 
만약 현재 프로젝트의 규모에서 오버엔지니어링이라고 판단된다면, 최소한의 비용으로 최대의 효과를 낼 수 있는 더 가볍고 프래그머틱(Pragmatic)한 대안을 제시하는 것이 우선입니다. 이것은 실험적인 프로젝트로 시작하지만 실제 상용 비즈니스로 발전할수 있는 프로젝트이기 때문에 추후 마이그레이션 비용도 고려해야 합니다. 예를들어 효율만을 고집하기 위해 flask를 사용하는 것을 지양해야 합니다.

## 🎯 핵심 과제 제안 (※ 오퍼스의 자율적인 판단 하에 변경/최적화 가능)

### 1. 빌드 캐싱 및 모노레포 오케스트레이션 고도화
- **제안 배경**: 현재 오로지 스크립트 래퍼 도구인 `Makefile`에만 과하게 의존하고 있어, 코드 변경분에 따른 스마트한 캐싱과 패키지 간 의존성 파이프라인 제어가 불가합니다.
- **구체적인 방향 제안 (예시)**: 
  - `packages/`를 신설하여 공통 설정 파일(eslint config, tsconfig 등) 혹은 공용 UI 컴포넌트, 공용 유틸 등을 구조적으로 분리
  - Turborepo, Nx, 또는 pnpm workspaces 중 **가장 설정이 직관적이고 유지보수가 용이한 도구**를 선정하여 도입
  - (선택/추가 제안) 관리 복잡성을 과하게 높이지 않는 선에서, API의 OpenAPI명세를 기반으로 프론트엔드의 DTO 인터페이스를 자동 생성하는(`openapi-typescript` 등 활용) 파이프라인 추가 여부 검토

### 2. API Gateway / Reverse Proxy 네트워크 계층 도입
- **제안 배경**: 현재 8080(Backend)과 5173(Frontend) 포트가 각각 독립적으로 직접 노출되어 통신하고 있어 운영 환경 배포 시 라우팅에 난항이 예상됩니다.
- **구체적인 방향 제안 (예시)**: 
  - `docker-compose.yml` 리팩토링 진행
  - Nginx, Traefik, Caddy 중 **오버엔지니어링 없이 가장 적은 설정 파일 변경으로 도입 가능한 도구**를 고려해 구성
  - `/api/*` 요청은 백엔드 컨테이너로, 나머지 `/` 요청은 SSR/CSR 서버로 라우팅 처리

### 3. 도메인 이벤트 기반 결합도 완화 및 메시지 큐(Message Queue) 설계
- **제안 배경**: 향후 `Notification`(알림), `Media`(미디어) 로직을 분리할 대비가 필요하지만 현재는 거대한 Monolith 내에서 서비스 클래스들이 강결합 되어 있습니다.
- **구체적인 방향 제안 (예시)**: 
  - 무작정 외부 인프라(Kafka, RabbitMQ 등)를 끼워넣어 **운영 복잡도를 비약적으로 높이기보다는**, 코드 레벨의 `Spring ApplicationEventPublisher`를 우선 활용해 도메인 이벤트를 격리
  - 추가적인 브로커가 꼭 필요하다면 **이미 구성되어 있는 Redis(Pub/Sub, Stream)**를 재활용하여 인프라 증설 없이 비동기 구조를 확보 (프로젝트 규모 대비 가성비/유지보수성 판단)

### 4. CI 원격 검증 자동화 파이프라인 (GitHub Actions 등)
- **제안 배경**: 로컬 검증기인 `make verify`는 잘 구성되어 있으나, PR(Pull Request) 병합 전 원격 파이프라인 구성이 없습니다.
- **구체적인 방향 제안 (예시)**: 
  - `.github/workflows/ci.yml` 작성
  - Gradle 의존성 및 npm 모듈을 캐싱하여 CI 런타임 최적화 시키기
  - 단순하고 관리하기 쉬운 형태로 백엔드와 프론트엔드 검증 단계를 분리/병렬화 적용

## 🛠 실행 프로토콜 (Action Rule)
1. **분석 및 독립적 설계 제시**: 이 문서에 적힌 아이디어를 억지로 따르지 마세요. 당신의 관점에서 "복잡도 증가와 유지보수성"을 철저히 고려하여 가장 프래그머틱한 기술을 선정하고, "구체적으로 어떤 파일 단위로 수정할 것인지(Phase 1~4 구체적 계획서)"를 아키텍처 설계안으로 출력(브리핑)해 줘.
2. **동의 후 실행**: 내가 설계안을 읽고 승인(`진행해` 등)하면, 그때부터 Phase 단위로 순차적인 코드베이스 적용(수정)을 시작해 줘.
3. **간결한 커뮤니케이션**: 장황한 서론 생략, 즉각 "당신의 분석에 기반한 아키텍처 방향성과 1단계 실행 계획"으로 답해 줘.
